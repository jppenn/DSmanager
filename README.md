# Dropship Manager

A modern web application for managing dropship orders sent to outside vendors. Internal staff create customer orders, assign line items to vendor purchase orders, email POs to vendors, track confirmations and shipping, and keep a full audit history. Vendors log into a dedicated portal to confirm POs and post tracking updates.

## Stack

- **Next.js 15** (App Router, React Server Components, Server Actions) + TypeScript
- **Supabase** — Postgres, Auth, Storage, Row Level Security
- **Resend** — transactional email (PO emails + alert digests)
- **Tailwind CSS v4** with a lightweight shadcn-style component library
- **Vercel** — hosting + Cron (hourly alert evaluation)

## Roles

| Role | Access |
| --- | --- |
| `admin` | Everything, including user management |
| `order_manager` | Customers, vendors, orders, POs, tracking, reports |
| `vendor` | Their own POs only (confirm, update status, add tracking + proof of delivery) |

Internal users (admin, order_manager) sign in with **magic links**. Vendor users have **password accounts** and use the **vendor portal** at `/portal`. Access is enforced both in the app (route guards in `src/lib/auth.ts`) and in the database (RLS policies).

## Project structure

```
src/
  app/
    (auth)/login            Magic-link + vendor password sign-in
    (app)/                  Internal app (dashboard, orders, POs, customers, vendors, reports, settings)
    (vendor)/portal         Vendor portal
    confirm/[token]         Public PO confirmation landing page
    api/cron/alerts         Vercel Cron endpoint (CRON_SECRET protected)

  components/               UI primitives + shared widgets
  lib/
    supabase/               Browser, server, admin (service role), and middleware clients
    types/database.ts       Hand-written typed schema (incl. relationships)

    email/                  Resend client + PO email template

  seed.sql                  Demo customers + vendors
```

## Database tables

`users`, `customers`, `vendors`, `customer_orders`, `customer_order_items`, `vendor_purchase_orders`, `vendor_purchase_order_items`, `shipments`, `order_notes`, `audit_events`, `email_logs`, plus an `order_financials` view for margin reporting.

Human-friendly identifiers (`SO-001000`, `PO-001000`) are generated by Postgres sequences. Every customer order and vendor PO has a complete timeline in `audit_events` (creation, status changes, emails, confirmations, tracking updates, notes).

## Local setup

1. **Install dependencies**

   ```bash
   npm install
   ```

2. **Create a Supabase project**, then run the SQL files in order in the Supabase SQL editor:
   - `supabase/migrations/0001_init.sql`
   - `supabase/migrations/0002_storage.sql`
   - `supabase/seed.sql` (optional demo data)

3. **Configure environment variables** — copy `.env.example` to `.env.local` and fill in:

   ```
   NEXT_PUBLIC_SUPABASE_URL=
   NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=
   SUPABASE_SECRET_KEY=
   RESEND_API_KEY=
   RESEND_FROM_EMAIL="Dropship Manager <orders@yourdomain.com>"
   NEXT_PUBLIC_APP_URL=http://localhost:3000
   CRON_SECRET=<long-random-string>
   ```

4. **Run the dev server**

   ```bash
   npm run dev
   ```

5. **Create the first admin.** Sign in once via magic link, then in the Supabase SQL editor:

   ```sql
   update public.users set role = 'admin' where email = 'you@company.com';
   ```

   After that you can invite other staff and vendor users from **Settings → Users**.

## How vendor accounts work

Invite a vendor from **Settings → Users** (role `vendor` + select their vendor company). They receive an email invite, set a password, and land in `/portal` where they only see their own POs. The trigger `handle_new_auth_user` auto-creates a profile row for every new auth user.

## Emailing a PO

Open a vendor PO and click **Send to vendor**. The email includes the PO number, ship-to address, line items, quantities, requested ship date, special instructions, and a tokenized confirmation link (`/confirm/<token>`). If `RESEND_API_KEY` is not set the send is logged to `email_logs` without dispatching, so the flow works in local dev.

## Alerts

`src/lib/alerts.ts` evaluates: not confirmed within 24h, ship date passed, tracking missing after ship, partially shipped, backordered, and cancelled. The dashboard renders them live; the Vercel Cron job (`vercel.json`, hourly) hits `/api/cron/alerts` to email an admin digest. The endpoint requires `Authorization: Bearer $CRON_SECRET`.

## Deploy to Vercel

1. Push to a Git repo and import it into Vercel.
2. Add all environment variables from `.env.example` in the Vercel project settings.
3. The `vercel.json` cron schedule runs `/api/cron/alerts` hourly. Set `CRON_SECRET` to match.
4. In Supabase Auth settings, add your production URL to the allowed redirect URLs (`https://yourapp.vercel.app/auth/callback`).

## Notes & trade-offs

- The typed Supabase schema (`src/lib/types/database.ts`) is hand-written rather than generated, including relationship metadata so embedded selects (`select("*, vendors(name)")`) are fully typed. Regenerate with `supabase gen types` if you prefer.
- The PO email template is plain HTML (in `src/lib/email/po-template.ts`) for maximum email-client compatibility; swap in `@react-email/components` if you want JSX templates.
- Vendor-facing views deliberately omit customer sell price / gross margin.
